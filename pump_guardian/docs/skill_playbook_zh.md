# 工业智能体「技能（Skill）」是什么？

## 技能包结构（与本 Demo 一一对应）

### `SKILL.md`（核心指令）

使用 Markdown 编写，说明**何时应触发**该技能、输入输出约定、与哪些工具协作。  
示例意图：*“当振动速度有效值（RMS）持续升高且伴随轴承频段能量集中时，启动轴承诊断技能。”*

### `scripts/`（执行逻辑）

可执行的 Python / TypeScript 等代码，用于**确定性计算**与**本地工具调用**。  
本项目的 `diag_tool.py` 负责：RMS、FFT、主频提取——这是通用大模型**无法凭空可靠替代**的部分。

### `references/`（参考资源）

PDF 手册、Excel 故障码表、JSON 规程（如 **ISO 10816**）等。  
智能体在需要判级、引用阈值时**随用随读**，避免把整本手册一次性塞进上下文。

### `assets/`（输出模板）

标准化工单、安环报告等模板，保证输出**符合企业版式与审计要求**。  
本 Demo 的 `wo_template.md` 即此类资产。

## 渐进式披露（Progressive Disclosure）与效率

传统做法常把“所有可能知识”写进超长提示词，导致：**响应慢、成本高、难维护**。  
技能化架构的策略是：

1. **启动阶段**只加载技能**元数据**（名称、能力摘要、触发条件关键词）。
2. 当用户请求或告警**命中**某技能时，再按需加载：`SKILL.md` → 脚本 → 参考 → 模板。

在处理大规模工业数据（如超大日志、万行级传感器记录）时，通过 **Code Mode / 工具执行** 把重计算放在本地或受控运行时，可显著降低**无效 Token 消耗**；业界实践里，将“全量上下文塞模型”改为“元数据 + 按需加载 + 代码执行”，往往能带来数量级的成本与延迟改善（具体比例因场景而异）。

## 工业落地：不止“会不会”，还要“靠不靠谱、划不划算”

对工业现场而言，智能体的评价通常是双重的：

1. **有效性（Effectiveness）**：能否在约束条件下**稳定解决问题**（可复现、可追溯、可审计）。
2. **效率（Efficiency）**：是否降低综合成本——时间、算力、人工协同与停机风险。

麦肯锡、德勤等 2024–2025 的多份调研都指向同一个现实：**大量组织已使用 AI，但只有少数通过流程再造获得显著业务结果**。  
技能化/agentic 工作流的价值，往往来自把“偶发演示”变成“可运营系统”：**标准、工具、模板、证据链**齐全，才能同时满足工程与管理两套 KPI。

> *It’s not just about capability — it’s about whether it can deliver results reliably and at a reasonable cost.*
